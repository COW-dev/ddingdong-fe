# PR 설명 작성

`.github/pull_request_template.md`에 정의된 형식을 그대로 따른다.

## 제목

대괄호 태그로 시작한다 (커밋 type과 다른 별도 목록):

```
[FEAT]     새로운 기능 추가
[FIX]      버그 수정
[HOTFIX]   긴급 버그 수정
[DOCS]     문서 추가/수정
[STYLE]    코드 스타일 변경 (포맷, 세미콜론 등)
[REFACTOR] 코드 리팩토링
[TEST]     테스트 코드 추가/수정
[CHORE]    빌드, 설정, 패키지 관리 등 기타 작업
```

예: `[FEAT] 로그인 API 추가`

## 본문 (섹션과 이모지까지 템플릿 그대로)

```markdown
## 🔥 연관 이슈

- close #이슈번호

## 🚀 작업 내용

## 🤔 고민했던 내용

## 💬 리뷰 중점사항
```

- `연관 이슈`: 브랜치명이나 커밋 메시지에서 이슈 번호를 찾을 수 있으면 채우고, 없으면 사용자에게 물어본다.
- `작업 내용`: `git log <base>..HEAD`와 `git diff <base>...HEAD`를 보고 실제 변경 사항을 요약한다.
- `고민했던 내용`, `리뷰 중점사항`: 커밋 로그만으로 알 수 없는 부분(트레이드오프, 애매한 결정)은 채우지 말고 빈 채로 두거나 사용자에게 물어본다 — 추측해서 채우지 않는다.

## 절차

1. `git status`, `git diff`, base 브랜치 대비 `git log`로 변경 범위를 파악한다.
2. 위 형식으로 제목/본문 초안을 만들어 사용자에게 보여준다.
3. 사용자 확인 후에만 `gh pr create`로 실제 PR을 생성한다 (push, PR 생성처럼 외부에 영향을 주는 작업은 항상 먼저 확인받는다).
